# Thesis: custom exercises should converge, not multiply

**Status:** design proposal (no code yet)
**Scope:** how a user adds their own exercises and maintains them over time
**One-line:** the same movement should resolve to *one* exercise no matter how it's typed, and be cheap to fix when it doesn't.

---

## 1. The problem in one sentence

Every path that touches a custom exercise keys on a **verbatim, exact id** derived
from the raw typed name, while recognition is **exact-id-only** and the fast
matcher **can't see custom exercises at all** — so the same movement fragments
into duplicates across history, records, and PRs, with no way to merge them back.

Type the same lift three ways and you get three exercises:

| You typed | Stored id | Result |
|---|---|---|
| `incline db press` | `incline-db-press` | new custom exercise #1 |
| `incline dumbbell press` | `incline-dumbbell-press` | new custom exercise #2 |
| `Incline Press (Dumbbells)` | `incline-press-dumbbells` | new custom exercise #3 |

…each with its own `exercise_records` row and its own history/PR line. This is the
core failure mode, and it compounds silently over months of logging.

## 2. Ground truth (how it works today)

- **Type:** `CustomExercise extends Workout` — structurally identical to a catalog
  entry plus `isCustom: true` + `createdAt`. Lives in AsyncStorage key
  `custom_exercises`; the catalog is a bundled read-only `exercises.json` (68 lifts).
  There is **no unified store** — three overlapping resolvers (`getWorkoutById`,
  `getExerciseById`, `getWorkoutByIdWithCustom`) each merge catalog+custom differently.
- **Creation, two paths:**
  1. *Implicit (primary):* logging an unrecognized name auto-creates a persisted
     custom exercise on finish (`toGeneratedWorkoutWithCustomExercises`), with
     muscle/category metadata generated by AI.
  2. *Explicit form:* Profile → Custom Exercises. Collects **only name + one
     equipment chip**; muscles/category/description are AI-filled, never user-edited.
- **Two matchers that disagree:**
  - *Live preview / offline* (`matchExerciseByName`) — fuzzy, abbrev-aware,
    equipment-aware, but indexes the **catalog only**. Custom exercises are invisible
    to it. (Being fixed separately as the autocomplete adjustment.)
  - *Commit / finish* (`matchExercise`) — sees custom exercises, but only by
    **exact kebab-id equality**.
- **Maintenance:** edit exposes name + equipment only. Rename recomputes the id and
  migrates it **in workout history but not in `exercise_records` or templates**
  (orphaned records). **No merge, no dedup** beyond exact-id upsert / exact-name check.
- **Stats:** strength percentile standards are a hardcoded catalog-id allowlist;
  custom exercises can never earn a tier (badges/PRs short-circuit on `isCustom`).
  That part is *correct* — there's no defensible standard for a bespoke movement.

### The eight concrete pain points

1. Duplicate proliferation (the headline).
2. Id generation is copy-pasted in **three** places that must stay in sync.
3. Three overlapping catalog+custom lookup resolvers; no single index.
4. No edit UI for muscles/category; tagging is brittle AI + keyword fallback that
   defaults unknowns to `chest`/`isolation`.
5. Equipment can silently desync from the display name.
6. `trackingType` left `undefined` on the AI-success path.
7. Percentile standards are catalog-only (fine, but nothing tells the user *why*
   their custom lift has no tier).
8. Rename orphans `exercise_records`.

## 3. Design principles

1. **One canonical identity per movement.** Dedupe on a *normalized* key, not the
   raw kebab id. Reuse the normalization the offline matcher already has
   (lowercase, strip equipment parens, expand abbreviations, singularize). The
   three "incline db press" spellings collapse to one canonical exercise.
2. **A single merged catalog+custom index.** One matcher powers preview, commit,
   and finish. (This is also the autocomplete fix — custom lifts become first-class
   in the fast local matcher.)
3. **Recognize-then-confirm, don't silently spawn.** When a typed name is close to
   an existing exercise (custom *or* catalog), surface "did you mean *Incline DB
   Press*?" in the live preview instead of quietly minting a near-duplicate on finish.
   Silent creation is the mechanism behind the duplicate pile-up.
4. **Maintenance is first-class.** Merge duplicates (re-point history + records +
   templates), edit muscle groups/category (not just name+equipment), and fix rename
   to migrate every reference.
5. **Degrade honestly on stats.** Custom exercises legitimately have no percentile
   tier; say so ("tracked by volume & PRs — no strength tier for custom lifts")
   instead of showing a silent 50th-percentile default.

## 4. Proposal (phased)

### Phase 0 — canonical identity + unified index *(foundational; fixes #1–#3)*
- Add a single `canonicalKey(name)` in one place; delete the three copy-pasted
  id-generators (have them delegate).
- Build **one** merged exercise index (catalog + custom) that both matchers consume,
  keyed by canonical key. Custom exercises become visible to the offline/preview
  matcher (the autocomplete adjustment falls out for free).
- On create/log, resolve against the canonical key **before** minting: an existing
  custom (or catalog) match reuses it instead of creating a duplicate.
- *Outcome:* the three-spellings case resolves to one exercise. No new UI.

### Phase 1 — maintenance UI *(fixes #4, #8; addresses existing dupes)*
- In the Custom Exercises manager: **merge** two exercises (pick survivor, re-point
  history + `exercise_records` + templates, delete the loser); **edit muscles/
  category** with a simple picker (seeded by the AI guess, now user-correctable).
- Fix `migrateExerciseId` to cover records + templates, not just history.
- Optional: a "these look like the same lift" nudge listing canonical-key collisions
  among existing customs, one-tap merge.

### Phase 2 — recognize-then-confirm *(fixes #1 at the source; UX)*
- When the preview's canonical match is a *near* (not exact) hit, show it as a
  suggestion chip the user taps to accept, rather than auto-creating on finish.
- Reserve silent creation for genuinely novel names (no near match).

### Small correctness fixes (fold into Phase 0/1)
- Always set `trackingType` on AI-created customs (#6).
- Re-derive equipment/display-name consistently on manual add (#5).
- Show the "no strength tier for custom lifts" affordance (#7).

## 5. Non-goals / open questions

- **Not** adding strength standards for custom lifts — no defensible percentile.
- **Migration of existing duplicates:** Phase 1's merge tool is manual; do we also
  want a one-time canonical-key auto-merge pass on upgrade? (Risk: wrongly merging
  two lifts a user deliberately kept distinct — favor the manual nudge.)
- **Where canonical normalization lives:** it should be the *same* module the
  offline matcher uses, so preview/commit/dedupe can't drift apart.
- **Aggressiveness of near-match confirm:** too eager and it nags; too loose and
  dupes slip through. Needs a tuned threshold (reuse the fuzzy budget, but require
  confirmation rather than auto-snapping).

## 6. Why this order

Phase 0 kills the root cause (fragmentation) with no UI and simultaneously delivers
the autocomplete win. Phase 1 lets users clean up the mess they already have. Phase
2 stops new dupes at the source. Each phase is independently shippable and testable,
and none of them touch the (correct) decision to keep percentile tiers catalog-only.
